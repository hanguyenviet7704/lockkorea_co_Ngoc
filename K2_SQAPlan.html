Part 1: Improvement of Specification using McCall's Quality Model

This section analyzes the project specifications using McCall's software quality factors. We have prioritized factors critical to the LockerKorea e-commerce platform and improved the specifications accordingly.

1. Product Operation Factors

These factors deal with the quality of the software while it is running.

  • Correctness (Essential):
    • Specification Improvement: The system must calculate order totals (including tax, shipping, and vouchers) with 100% accuracy. Payment gateway integration (Stripe) must correctly record transaction statuses.
    • Justification: Financial discrepancies lead to direct revenue loss and loss of customer trust.
  • Reliability (Essential):
    • Specification Improvement: The checkout process must have a success rate of 99.9%. If a payment fails, the user must be clearly informed, and the cart contents must be preserved.
    • Justification: A failed checkout is a lost sale.
  • Integrity (Essential):
    • Specification Improvement: Customer personal data (Address, Phone) and Order History must be accessible only to the owner and authorized Admin staff. API endpoints must be secured with JWT tokens.
    • Justification: E-commerce sites are prime targets for data theft.
  • Usability (High Priority):
    • Specification Improvement: The "Add to Cart" to "Checkout" flow must be completed in under 4 steps. The site must be fully responsive on mobile devices (375px+ width).
    • Justification: Friction in the UI directly increases cart abandonment rates.

2. Product Revision Factors

These factors deal with the quality of the software's ability to evolve.

  • Maintainability (Medium Priority):
    • Specification Improvement: The Product and Order modules in the Backend must be decoupled. Frontend components (e.g., ProductCard) should be reusable across the Home, Category, and Search pages.
    • Justification: E-commerce rules change frequently (e.g., new discount types), requiring easy code updates.
  • Testability (High Priority):
    • Specification Improvement: The pricing logic (Voucher application) must be isolated in pure functions to allow for comprehensive unit testing.

3. Product Transition Factors

These factors deal with the software's adaptability to new environments.

  • Interoperability (High Priority):
    • Specification Improvement: The system must integrate seamlessly with the Stripe Payment Gateway and Google Login (OAuth 2.0).
    • Justification: Modern e-commerce relies on external services for payments and identity.

4. Justification for Unused / Lower Priority Factors

The following factors from McCall's model were considered but deemed lower priority or "not used" for this specific release:

  • Portability:
    • Status: Not Used / Low Priority.
    • Explanation: The application is a Web App designed to run in standard browsers (Chrome, Edge). We are not deploying the software to different hardware architectures (embedded, mainframes). Browser compatibility takes precedence over defined "Portability".
  • Reusability:
    • Status: Low Priority.
    • Explanation: While we use modular components, the goal is to build a specific brand implementation (LockerKorea). We are not building a generic "Shopify-like" SaaS platform to sell to other vendors, so determining the reusability of the entire system for other scopes is not a primary objective.
  • Flexibility:
    • Status: Low Priority.
    • Explanation: The system is optimized for a specific catalog structure and order flow. Making the schema "flexible" enough to sell any type of service (e.g., subscriptions, digital downloads) would overcomplicate the database design for the initial release, which focuses on physical goods.

Part 2: Software Quality Assurance Plan (SQAP)

Based on IEEE Std 730-2014, Annex C

1. Purpose

The purpose of this Software Quality Assurance Plan (SQAP) is to define the activities, standards, tools, and procedures required to ensure that the LockerKorea e-commerce application meets its established requirements. This plan serves as a roadmap for Group K2 to maintain high quality throughout the development lifecycle.

2. Reference Documents

  • IEEE Std 730™-2014, IEEE Standard for Software Quality Assurance.
  • McCall, J.A., Factors in Software Quality, 1977.
  • LockerKorea Project Requirements Specification (SRS).

3. Management

3.1 Organization

The SQA role adheres to the principle of objectivity.

  • SQA Lead: Independent of the coding tasks, responsible for auditing the process.
  • Project Manager: Responsible for schedule and resources.
  • Developers: Responsible for Unit Testing and Peer Reviews.

3.2 Tasks

  • Review: Conduct formal reviews of the SRS and Design documents.
  • Audit: Check that code follows the defined Angular and Spring Boot style guides.
  • Test: Execute System and Acceptance tests.

3.3 Roles and Responsibilities

  • SQA Lead: Creates this plan, maintains SQA records, and has the authority to block a release if critical bugs exist.
  • Development Team: Fixes defects identified by SQA and writes automated tests.

4. Documentation

The project shall produce the following documents, which are subject to SQA review:

  1. Software Requirements Specification (SRS): Defines features (Product Management, Cart, Checkout).
  2. Software Design Description (SDD): Describes the architecture (Angular Frontend, Spring Boot Backend) and lockerkorea database schema.
  3. Source Code: The actual implementation in the Git repository.
  4. Test Case Specifications: Detailed steps for verifying each requirement (e.g., "TC-01: Add item to cart").
  5. Test Reports: Results of test execution.

5. Standards, Practices, Conventions, and Metrics

5.1 Standards

  • Coding Standard:
    • Frontend: Angular Style Guide (strict TypeScript mode).
    • Backend: Google Java Style Guide.
  • Documentation: All documents must follow the group's Markdown template.

5.2 Metrics

SQA will collect the following metrics to evaluate quality:

  • Defect Density: Number of defects per 1,000 lines of code.
  • Test Coverage: % of code lines executed by automated tests (Target: >70%).
  • Performance: Average page load time < 2 seconds.

6. Software Reviews

Reviews are the primary method for ensuring quality before testing begins.

  • Requirement Review: Validate that requirements are "SMART" (e.g., "Vouchers must apply only to specific categories").
  • Design Review: Verify that the database schema supports all storage needs (Orders, OrderDetails, Payments).
  • Code Review (Peer Review):
    • Every Pull Request (PR) must be reviewed by at least one other team member.
    • Reviewers check for logic errors, security flaws (e.g., SQL injection), and style violations.

7. Test

The testing strategy complies with the McCall factors identified in Part 1.

  • Unit Testing (White Box): Developers write tests for backend services (e.g., OrderService).
  • Integration Testing: Verify communication between the Angular Frontend and Spring Boot Backend APIs.
  • System Testing (Black Box): Integrated testing of complete user scenarios (e.g., "Guest user buys a product").
  • Acceptance Testing: Final verification against the original requirements checklist.

8. Problem Reporting and Corrective Action

  • Defect Logging: All defects are logged in the Issue Tracker (e.g., GitHub Issues).
  • Format: Summary, Steps to Reproduce, Expected Result, Actual Result, Severity.
  • Resolution Process:
    1. SQA logs the bug.
    2. Dev analyzes and fixes.
    3. SQA re-tests the fix ("Regression Testing").
    4. Ticket is Closed.

9. Tools, Techniques, and Methodologies

  • Version Control: Git / GitHub.
  • CI/CD: GitHub Actions (automatically runs builds on every push).
  • Analysis Tools: ESLint (Frontend), IDE-based linting (Backend).
  • Testing Tools: Postman (API), Chrome DevTools.

10. Media Control

  • Source Code Management: Main branch is protected. Direct pushes are disabled; changes require a Pull Request.
  • Backups: Daily snapshots of the lockerkorea database.

11. Supplier Control

  • This project does not sub-contract development.
  • Third-party Libraries: Libraries like stripe-java or angular-material are vetted for security vulnerabilities.

12. Records Collection, Maintenance, and Retention

  • SQA Records (Test Plans, Review Logs) are stored in the project repository.
  • Records are retained for the duration of the course.

13. Training

  • All team members must read the Project Setup Guide (README.md) to understand the Docker environment and build process.

14. Risk Management

  • Risk: Payment gateway sandbox failure.
    • Mitigation: Implement a "Mock Payment" mode for local development.
  • Risk: Data loss during container reset.
    • Mitigation: Use Docker volumes to persist MySQL data sections.
